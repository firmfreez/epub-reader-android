# EPUB Reader

### Приложение уже находится на стадии модерации в RuStore

## Описание

Android-приложение-читалка для локального чтения учебной литературы в формате EPUB.

Приложение работает **полностью офлайн**: пользователь выбирает EPUB-файл, приложение импортирует его, сохраняет метаданные и прогресс чтения, после чего позволяет открывать книгу, читать содержимое, переходить между разделами и продолжать чтение с последней позиции.

В рамках задания реализован полный пользовательский сценарий:

- импорт EPUB-файла с устройства;
- локальное хранение метаданных в БД;
- отображение текста и изображений из EPUB;
- навигация по документу;
- сохранение и восстановление позиции чтения;
- работа без доступа к сети;
- обработка ошибок и нестандартных сценариев.

## Архитектура

### Общий подход

Приложение построено на сочетании **MVVM + MVI** и модульной архитектуры.

- **MVVM** используется как базовая модель взаимодействия между UI и слоем представления.
- **MVI** применяется для управления состоянием экранов: экран получает `UiState`, пользовательские действия отправляются в `Action`, а одноразовые эффекты обрабатываются через `UiEvent`.

Такой подход даёт:
- предсказуемый поток данных;
- явное разделение ответственности;
- удобство поддержки и развития;
- устойчивость к усложнению бизнес-логики;
- более простой контроль состояний загрузки, ошибок и навигационных эффектов.

Архитектура проекта изначально спроектирована как модульная, чтобы:
- уменьшить связанность между частями приложения;
- локализовать ответственность;
- упростить замену реализаций;
- подготовить основу для возможного масштабирования и частичного переноса на KMP в будущем.

---

### Архитектурные принципы

В проекте используются следующие принципы:

- UI не работает напрямую с хранилищами, системными API и низкоуровневой логикой;
- бизнес-логика инкапсулирована в `UseCase`;
- зависимости передаются через DI;
- состояние экрана централизовано во `ViewModel`;
- общая и проектная логика разделены по отдельным модулям;
- платформенно-зависимые реализации по возможности изолируются от абстракций.

## Организация проекта по модулям

### build-logic / Convention Plugins

В проекте настроены **Convention Plugins**, которые выносят общую build-конфигурацию из модулей в единое место.

Это позволяет:
- не дублировать настройки Gradle между модулями;
- централизованно подключать плагины, зависимости и build-конфигурацию;
- упростить поддержку проекта;
- сделать подключение новых модулей более быстрым и предсказуемым.

По сути, convention plugins формируют единый стандарт сборки внутри проекта.

---

### buildSrc

`buildSrc` используется как дополнительный слой для build-инфраструктуры проекта.

Основная задача — не подключать плагины и общие build-элементы по строковым литералам, а использовать заранее описанные константы и обёртки.

Это даёт:
- меньше ошибок из-за опечаток;
- более удобную навигацию по build-коду;
- более читаемую конфигурацию Gradle;
- единообразие подключения плагинов и зависимостей.

## common-модули

`common` — это уже проектно-специфичный, но всё ещё общий слой.

В отличие от `core`, эти модули знают о текущем приложении и решают общие для него задачи.

### common:ui
Содержит общую UI-логику проекта:
- `BaseViewModel`;
- базовые мапперы;
- общие механизмы обработки состояний, событий и ошибок.

### common:ui-components
Содержит общие Compose-компоненты, которые переиспользуются между экранами.

### common:domain
Общий доменный слой проекта:
- domain-модели;
- контракты;
- сущности, которые нужны в разных частях приложения.

### common:data
Инициализация и общая data-конфигурация проекта.

### common:usecases
Общие `UseCase`, которые используются сразу в нескольких местах приложения.

Технически часть этой логики можно было бы разместить внутри `common:domain`, но выделение `usecases` в отдельный модуль даёт более понятную организацию проекта и упрощает навигацию по коду.

## feature-модули

`feature` — это модули, отвечающие за конкретные пользовательские сценарии и экраны.

На текущем этапе они разделены **поэкранно**. В приложении реализовано 3 основные фичи:

- `splash` — стартовый экран, где может происходить первоначальная подготовка приложения;
- `home` — экран библиотеки книг;
- `reader` — экран чтения книги.

Для каждой фичи используются модули:

- **api** — публичный контракт фичи;
- **impl** — конкретная реализация.

### Зачем разделение на api / impl
Такое разделение позволяет:
- не тянуть реализацию туда, где достаточно контракта;
- уменьшать связанность между модулями;
- проще заменять реализации;
- закладывать основу под мультиплатформенность.

В текущем проекте для feature-модулей не создавались отдельные `data` и `domain` модули, потому что фичи пока небольшие и такая декомпозиция была бы избыточной.

На текущем этапе достаточно:
- использовать общие `common`-модули;
- либо локализовать простую вспомогательную data-логику внутри `impl`.

Пример: в `reader` отдельный `data`-модуль не создавался, так как локальное хранение открытых `Publication` достаточно компактно и реализовано прямо внутри `impl`.

## lib-модули

`lib` — это отдельные инфраструктурные или библиотечные модули, не являющиеся пользовательскими фичами, но предоставляющие важную функциональность приложению.

### lib:book-importer
Отвечает за:
- анализ EPUB-файла;
- извлечение метаданных;
- копирование обложек в кэш;
- подготовку данных к импорту.

### lib:crash-reporter
Содержит интеграцию с Crashlytics и отвечает за сбор и отправку crash-отчётов.

### lib:di
Хранит DI-specific сущности, которые не относятся напрямую к бизнес-логике приложения.

В текущем проекте это, например:
- qualifiers для `CoroutineDispatcher`.

Для `di` используется только `domain`-часть, так как сам модуль не требует разделения на api / impl.

### lib:navigation
Отвечает за навигационную абстракцию.

- в `api` находятся навигационные ключи и контракты;
- в `impl` — логика, которая по этим ключам возвращает конкретный экран.

Такой подход изолирует навигационную реализацию и не даёт фичам зависеть друг от друга напрямую.

Текущая структура проекта даёт несколько преимуществ:

- код проще поддерживать и расширять;

## Слой представления и управление состоянием

Каждый экран работает через `ViewModel`.

Используется следующая схема:

- UI подписывается на `UiState`;
- пользовательские действия отправляются во `ViewModel` через `Action`;
- `ViewModel` изменяет состояние;
- одноразовые эффекты, такие как навигация или команды для reader, отправляются через `UiEvent`.

Такой подход позволяет отделить:
- **состояние** экрана;
- **действия** пользователя;
- **одноразовые эффекты**.

Это особенно полезно для экранов с более сложной логикой, например для `reader`, где нужно отдельно управлять:
- текущей книгой;
- позицией чтения;
- состоянием загрузки;
- отображением overlay;
- навигацией по документу;
- сохранением прогресса.

## Локальное хранение
В БД сохраняются:
- идентификатор книги;
- название;
- автор;
- описание;
- путь к книге;
- путь к локальной обложке;
- размер файла;
- контрольная сумма файла;
- прогресс чтения;
- последний `Locator` в JSON-виде;
- время добавления.

### Восстановление позиции чтения
Позиция сохраняется через `Readium Locator`.

Почему именно `Locator`:
- он является нативным способом хранения позиции чтения для EPUB;
- не зависит от “экранных страниц”;
- корректно восстанавливает место чтения даже при сложной структуре книги.

При повторном открытии книги:
- из БД читается сохранённый `Locator`;
- он передаётся в Readium как `initialLocator`;
- чтение продолжается с последней позиции.

## Обработка ошибок и нестандартных сценариев

Поддерживаются следующие сценарии:

- выбран не EPUB;
- файл недоступен;
- доступ к `Uri` потерян;
- EPUB повреждён;
- не удалось извлечь `Publication`;
- не удалось восстановить позицию;
- книга уже добавлена;
- пользователь открыл приложение из системы через EPUB-файл;
- файл был удалён или перемещён после добавления.

### Подход к ошибкам
- ошибки доменного уровня преобразуются в понятные UI-сообщения;
- неожиданные исключения не пробрасываются в UI напрямую;
- критические сбои логируются и отправляются в Crashlytics;
- пользователь получает контролируемое состояние ошибки, а не crash.

## Эксплуатация в production

### Мониторинг и поддержка
Используется:
- **Firebase Crashlytics** — для crash-отчётов;
- локальное логирование ключевых сценариев — импорт, открытие книги, ошибки EPUB, восстановление позиции.

### Как выявляются инциденты
- crash — через Crashlytics;
- некритические сценарии — через логи и воспроизведение;
- ошибки работы с EPUB — через пользовательские сообщения и технические логи.

### Приоритизация инцидентов
1. **Критичные**
    - crash при открытии приложения;
    - crash при открытии книги;
    - потеря данных о библиотеке или прогрессе.
2. **Высокий приоритет**
    - книга не открывается;
    - позиция не восстанавливается;
    - файл ошибочно определяется как битый или дублирующийся.
3. **Средний приоритет**
    - визуальные артефакты;
    - некритичные лаги;
    - проблемы с меню, состояниями, overlays.

### Процесс исправления
- воспроизведение проблемы;
- анализ логов и crash-отчётов;
- локализация причины;
- исправление;
- ручная проверка ключевых сценариев;
- выпуск новой версии.
